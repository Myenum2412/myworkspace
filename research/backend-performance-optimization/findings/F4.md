# F4: Async Processing & Background Jobs

## Findings

### [1] Kafka achieves 605 MB/s peak throughput with 5ms p99 latency under load, outperforming Pulsar (305 MB/s, 25ms p99) and RabbitMQ (38 MB/s, 1ms p99 at reduced load).
- quote: "Kafka delivers the best throughput while providing the lowest end-to-end latencies up to the p99.9th percentile."
- url: https://www.confluent.io/blog/kafka-fastest-messaging-system/
- source_type: primary (Confluent benchmarking study, Aug 2020)
- published: 2020-08-21
- confidence: high

### [2] AWS Step Functions Express Workflows support 100,000 executions per second with at-least-once semantics, while Standard Workflows support 2,000 executions/second with exactly-once semantics and run up to one year.
- quote: "Standard workflows have exactly-once workflow execution and can run for up to one year. Express workflows have at-least-once workflow execution and can run for up to five minutes."
- url: https://docs.aws.amazon.com/step-functions/latest/dg/welcome.html
- source_type: primary (AWS official documentation)
- published: unknown
- confidence: high

### [3] NATS JetStream achieves linearizable consistency for writes using a RAFT-based distributed quorum algorithm, providing immediate consistency rather than eventual consistency for stream operations.
- quote: "JetStream uses a NATS optimized RAFT distributed quorum algorithm to distribute the persistence service between NATS servers in a cluster while maintaining immediate consistency (as opposed to eventual consistency) even in the face of failures."
- url: https://docs.nats.io/nats-concepts/jetstream
- source_type: primary (NATS official documentation)
- published: unknown
- confidence: high

### [4] Celery 5.6 can process millions of tasks per minute with sub-millisecond round-trip latency when using RabbitMQ with librabbitmq and optimized settings.
- quote: "A single Celery process can process millions of tasks a minute, with sub-millisecond round-trip latency (using RabbitMQ, librabbitmq, and optimized settings)."
- url: https://docs.celeryq.dev/en/stable/getting-started/introduction.html
- source_type: primary (Celery official documentation)
- published: unknown
- confidence: high

### [5] Event-driven architecture eliminates continuous polling costs by using push-based event routing, reducing network bandwidth consumption, CPU utilization, and idle fleet capacity.
- quote: "Event-driven architectures are push-based, so everything happens on-demand as the event presents itself in the router. This way, you're not paying for continuous polling to check for an event."
- url: https://aws.amazon.com/event-driven-architecture/
- source_type: primary (AWS official documentation)
- published: unknown
- confidence: high

### [6] Kafka's replication protocol ensures durability without synchronous fsync by tracking what has been fsynced to disk, allowing it to leverage OS page cache batching for better performance.
- quote: "Kafka's replication protocol was carefully designed to ensure consistency and durability guarantees without the need for synchronous fsync by tracking what has been fsynced to the disk and what hasn't."
- url: https://www.confluent.io/blog/kafka-fastest-messaging-system/
- source_type: primary (Confluent benchmarking study)
- published: 2020-08-21
- confidence: high

### [7] Temporal workflow engine uses heartbeat-based failure detection where workers send periodic pings; if a heartbeat is not received within the Heartbeat Timeout, the Activity Task fails and retries occur.
- quote: "A Heartbeat Timeout is the maximum time between Activity Heartbeats. If this timeout is reached, the Activity Task fails and a retry occurs if a Retry Policy dictates it."
- url: https://docs.temporal.io/encyclopedia/detecting-activity-failures
- source_type: primary (Temporal official documentation)
- published: unknown
- confidence: high

### [8] Redis Streams consumer groups enable automatic claiming of idle pending messages from failed consumers via XAUTOCLAIM, providing built-in worker failover at O(1) complexity.
- quote: "XAUTOCLAIM - Practical pattern: Automatically claim idle pending messages using XAUTOCLAIM for simplified consumer failure recovery."
- url: https://redis.io/docs/latest/develop/data-types/streams/
- source_type: primary (Redis official documentation)
- published: unknown
- confidence: high

### [9] NATS JetStream pull consumers support horizontal scalability without partition management, where multiple consumers transparently share workload from a single stream.
- quote: "Pull consumers can and are meant to be shared between applications (just like queue groups) in order to provide easy and transparent horizontal scalability of the processing or consumption of messages in a stream without having to worry about having to define partitions."
- url: https://docs.nats.io/nats-concepts/jetstream
- source_type: primary (NATS official documentation)
- published: unknown
- confidence: high

### [10] RabbitMQ latency degrades significantly above 30 MB/s throughput in mirrored queue configuration, making it unsuitable for high-throughput workloads despite its excellent low-latency characteristics at lower loads.
- quote: "RabbitMQ latencies degrade significantly at throughputs higher than the 30 MB/s."
- url: https://www.confluent.io/blog/kafka-fastest-messaging-system/
- source_type: primary (Confluent benchmarking study)
- published: 2020-08-21
- confidence: high
