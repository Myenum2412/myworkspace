# F5: Performance Monitoring & Profiling

## Findings

### [1] OpenTelemetry is the vendor-neutral industry standard for observability, supported by over 90 vendors
- quote: "OpenTelemetry is an industry-standard supported by more than 90 observability vendors, integrated by many libraries, services, and apps"
- url: https://opentelemetry.io/docs/
- source_type: primary
- published: unknown
- confidence: high

### [2] Prometheus recommends instrumenting all library subsystems with query count, errors, and latency metrics
- quote: "If it is a library used to access some resource outside of the process (for example, network, disk, or IPC), track the overall query count, errors (if errors are possible) and latency at a minimum."
- url: https://prometheus.io/docs/practices/instrumentation/
- source_type: primary
- published: 2026
- confidence: high

### [3] Prometheus metrics label cardinality should stay below 10 for most metrics to avoid performance degradation
- quote: "As a general guideline, try to keep the cardinality of your metrics below 10, and for metrics that exceed that, aim to limit them to a handful across your whole system."
- url: https://prometheus.io/docs/practices/instrumentation/
- source_type: primary
- published: 2026
- confidence: high

### [4] gperftools provides CPU and heap profiling with near-zero overhead when dormant
- quote: "The cpu profiler, heap checker, and heap profiler will lie dormant, using no memory or CPU, until you turn them on. (Thus, there's no harm in linking -lprofiler into every application.)"
- url: https://github.com/gperftools/gperftools
- source_type: primary
- published: 2023 (last refresh Aug 2023)
- confidence: high

### [5] eBPF enables kernel-level performance tracing with native JIT-compiled execution speed
- quote: "The Just-in-Time (JIT) compilation step translates the generic bytecode of the program into the machine specific instruction set to optimize execution speed of the program. This makes eBPF programs run as efficiently as natively compiled kernel code."
- url: https://ebpf.io/what-is-ebpf/
- source_type: primary
- published: 2026
- confidence: high

### [6] Datadog APM provides thread-level distributed tracing correlated with infrastructure metrics and database queries
- quote: "Identify root causes quickly by correlating traces with logs, infrastructure metrics, database queries, network calls, and frontend telemetry—all in one view"
- url: https://www.datadoghq.com/product/apm/
- source_type: primary
- published: 2026
- confidence: high

### [7] Prometheus defines three service types for monitoring: online-serving, offline-processing, and batch jobs
- quote: "For monitoring purposes, services can generally be broken down into three types: online-serving, offline-processing, and batch jobs. There is overlap between them, but every service tends to fit well into one of these categories."
- url: https://prometheus.io/docs/practices/instrumentation/
- source_type: primary
- published: 2026
- confidence: high

### [8] eBPF tools like bcc and bpftrace provide application and system profiling through kernel hooks without code modification
- quote: "BCC is a framework that enables users to write python programs with eBPF programs embedded inside them. The framework is primarily targeted for use cases which involve application and system profiling/tracing."
- url: https://ebpf.io/what-is-ebpf/
- source_type: primary
- published: 2026
- confidence: high

### [9] A Java counter takes 12-17ns to increment depending on contention, establishing baseline instrumentation overhead
- quote: "A Java counter takes 12-17ns to increment depending on contention. Other languages will have similar performance."
- url: https://prometheus.io/docs/practices/instrumentation/
- source_type: primary
- published: 2026
- confidence: high

### [10] Datadog Continuous Profiler analyzes runtime and code inefficiencies in production via timeline view
- quote: "Diagnose runtime and code inefficiencies in production by using Continuous Profiler's timeline view"
- url: https://www.datadoghq.com/blog/continuous-profiler-timeline-view/
- source_type: primary
- published: 2026
- confidence: medium

### [11] gperftools includes tcmalloc, a high-performance multi-threaded malloc replacement for memory profiling
- quote: "gperftools is a collection of a high-performance multi-threaded malloc() implementation, plus some pretty nifty performance analysis tools."
- url: https://github.com/gperftools/gperftools
- source_type: primary
- published: 2023
- confidence: high

### [12] eBPF provides bounded loop verification and memory safety guarantees through its verifier
- quote: "The verifier will evaluate all possible execution paths and must be capable of completing the analysis within the limits of the configured upper complexity limit."
- url: https://ebpf.io/what-is-ebpf/
- source_type: primary
- published: 2026
- confidence: high
