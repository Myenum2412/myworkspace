# F1: API Response Time Optimization

## Findings

### [1] HTTP/2 multiplexing eliminates application-layer head-of-line blocking by enabling concurrent request/response streams over a single TCP connection
- quote: "HTTP/2 addresses these issues by defining an optimized mapping of HTTP's semantics to an underlying connection. Specifically, it allows interleaving of messages on the same connection and uses an efficient coding for HTTP fields."
- url: https://httpwg.org/specs/rfc9113.html
- source_type: primary
- published: 2022-06-01
- confidence: high

### [2] HTTP/2 uses HPACK field compression to eliminate redundant header overhead, allowing many requests to be compressed into one packet
- quote: "Because HTTP fields used in a connection can contain large amounts of redundant data, frames that contain them are compressed. This has especially advantageous impact upon request sizes in the common case, allowing many requests to be compressed into one packet."
- url: https://httpwg.org/specs/rfc9113.html
- source_type: primary
- published: 2022-06-01
- confidence: high

### [3] HTTP/2 and HTTP/3 prohibit Connection and Keep-Alive headers; persistent connections are managed internally by the protocol
- quote: "Connection-specific header fields such as Connection and Keep-Alive are prohibited in HTTP/2 and HTTP/3. Chrome and Firefox ignore them in HTTP/2 responses, but Safari conforms to the HTTP/2 specification requirements and does not load any response that contains them."
- url: https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Keep-Alive
- source_type: primary
- published: 2025-07-04
- confidence: high

### [4] HTTP/1.1 Keep-Alive header allows tuning idle timeout and max requests per connection to reduce connection setup overhead
- quote: "Keep-Alive: timeout=5, max=200" — the timeout parameter specifies seconds an idle connection remains open; max specifies the maximum number of requests before closing.
- url: https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Keep-Alive
- source_type: primary
- published: 2025-07-04
- confidence: high

### [5] HTTP/3 (QUIC over UDP) resolves TCP-level head-of-line blocking that persists even in HTTP/2, reducing latency on unreliable networks
- quote: "HTTP/3 addresses this limitation by using QUIC, a protocol built on UDP, instead of TCP. This change improves performance, reduces connection setup time, and enhances stability on degraded or unreliable networks."
- url: https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Messages
- source_type: primary
- published: 2025-09-02
- confidence: high

### [6] HTTP compression (gzip/Brotli) reduces typical JSON API payloads by 60-85% with zero changes to the API contract
- quote: "The most impactful optimization for most APIs is simply enabling HTTP compression (gzip or Brotli) — this alone reduces JSON payload size by 60-85% with zero changes to the API contract, zero client-side library requirements, and full debuggability."
- url: https://jsonic.io/guides/json-compression
- source_type: secondary
- published: 2026-05-20
- confidence: high

### [7] Brotli level 4 provides ~8% better compression than gzip level 6 at similar CPU cost, with ~2.4x faster decompression (~600 MB/s vs ~250 MB/s)
- quote: "Brotli level 4 provides ~8% better compression than gzip level 6 with similar CPU cost." / "Brotli decompresses at ~600 MB/s vs gzip ~250 MB/s — Brotli is faster to decompress"
- url: https://jsonic.io/guides/json-compression
- source_type: secondary
- published: 2026-05-20
- confidence: high

### [8] On a 500 KB JSON API response, Brotli level 4 compresses to 95 KB in 14 ms, while zstd level 3 reaches 97 KB in only 3 ms — the fastest real-time compression option
- quote: "Brotli level 4: 95 KB, 5.3x ratio, 14 ms compress, 2 ms decompress — sweet spot. zstd level 3: 97 KB, 5.1x ratio, 3 ms compress, 1 ms decompress — fastest."
- url: https://codelit.io/blog/api-response-compression
- source_type: secondary
- published: 2026-03-29
- confidence: medium

### [9] Compression should be applied at the API gateway layer for most architectures; per-service compression is only justified for large internal payloads
- quote: "Your API gateway, reverse proxy, or CDN handles compression. Services return uncompressed responses internally. Advantages: Services stay simple — no compression logic in application code. Single configuration point for all services. Gateway can cache compressed variants."
- url: https://codelit.io/blog/api-response-compression
- source_type: secondary
- published: 2026-03-29
- confidence: medium

### [10] Responses under 256 bytes should NOT be compressed — compression header overhead can make them larger than the original
- quote: "Compression adds a header overhead of 10-20 bytes. For tiny responses, the compressed output can be larger than the original. Most servers use a minimum threshold: gzip_min_length 256."
- url: https://codelit.io/blog/api-response-compression
- source_type: secondary
- published: 2026-03-29
- confidence: high

### [11] Zstandard (zstd, RFC 8878) achieves comparable compression ratios to Brotli while compressing 5-10x faster, making it ideal for server-to-server APIs and internal data pipelines
- quote: "Zstandard (zstd) offers the best speed-to-ratio tradeoff for server-to-server communication. Compression ratio: comparable to Brotli. Speed: dramatically faster compression than Brotli at similar ratios. Decompression speed: ~1200 MB/s — the fastest of all three."
- url: https://codelit.io/blog/api-response-compression
- source_type: secondary
- published: 2026-03-29
- confidence: medium

### [12] Gateway-level keep-alive tuning (idle timeout, max connections per host) is critical for high-throughput API proxies — KrakenD defaults to 15s dialer keep-alive and 250 max idle connections per host
- quote: "dialer_keep_alive: default 15s, max_idle_connections_per_host: default 250, idle_connection_timeout: duration an idle connection stays open."
- url: https://www.krakend.io/blog/keep-alive-optimizing-performance/
- source_type: secondary
- published: 2023-11-12
- confidence: medium
