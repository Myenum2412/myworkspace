# F3: Caching Architecture

## Findings

### [1] Cache-aside pattern invalidates cache on write and repopulates on the next read, creating a brief window of staleness between write completion and subsequent reads.
- quote: "The Cache-Aside pattern invalidates the cached entry on write and repopulates it on the next read. Between the write and the next read, a reader can experience a cache miss or briefly see stale data. This behavior distinguishes the Cache-Aside pattern from write-through caching, which updates the data store and the cache in the same write operation so that readers see the new value immediately after a successful write."
- url: https://learn.microsoft.com/en-us/azure/architecture/patterns/cache-aside
- source_type: primary
- published: 2025-09-11
- confidence: high

### [2] Multi-level caching with local private caches and shared distributed caches provides resilience: local caches act as a fallback buffer if the shared cache service is temporarily unavailable.
- quote: "Consider implementing a local, private cache in each instance of an application, together with the shared cache that all application instances access. When the application retrieves an item, it can check first in the local cache, then in the shared cache, and finally in the original data store. The local cache can be populated using the data in either the shared cache, or in the database if the shared cache is unavailable."
- url: https://learn.microsoft.com/en-us/azure/architecture/best-practices/caching
- source_type: primary
- published: 2026-03-25
- confidence: high

### [3] Private in-memory caches cause cross-instance inconsistency: each application instance holds its own snapshot of data at different points in time, so the same query can return different results across instances.
- quote: "If you have multiple instances of an application that uses this model running concurrently, each application instance has its own independent cache that holds its own copy of the data. Therefore, the same query performed by these instances can return different results."
- url: https://learn.microsoft.com/en-us/azure/architecture/best-practices/caching
- source_type: primary
- published: 2026-03-25
- confidence: high

### [4] Lazy loading (cache-aside) incurs a three-trip cache miss penalty: initial cache lookup, database query, then cache write — each adding network latency.
- quote: "There is a cache miss penalty. Each cache miss results in three trips: 1. Initial request for data from the cache. 2. Query of the database for the data. 3. Writing the data to the cache. These misses can cause a noticeable delay in data getting to the application."
- url: https://docs.aws.amazon.com/AmazonElastiCache/latest/red-ug/Strategies.html
- source_type: primary
- published: unknown
- confidence: high

### [5] Write-through caching guarantees non-stale cache data but introduces a two-trip write penalty and creates a missing-data problem when new nodes are provisioned without pre-populated cache entries.
- quote: "The advantages of write-through are as follows: Data in the cache is never stale. Because the data in the cache is updated every time it's written to the database, the data in the cache is always current. ... The disadvantages of write-through are as follows: Missing data. If you spin up a new node, whether due to a node failure or scaling out, there is missing data."
- url: https://docs.aws.amazon.com/AmazonElastiCache/latest/red-ug/Strategies.html
- source_type: primary
- published: unknown
- confidence: high

### [6] Combining write-through with TTL creates a hybrid strategy that ensures freshness while preventing cache churn from data that is never read, balancing the advantages of both lazy loading and write-through.
- quote: "By adding a time to live (TTL) value to each write, you can have the advantages of each strategy. At the same time, you can and largely avoid cluttering up the cache with extra data."
- url: https://docs.aws.amazon.com/AmazonElastiCache/latest/red-ug/Strategies.html
- source_type: primary
- published: unknown
- confidence: high

### [7] Write-through architectures require a durable outbox pattern to recover cache consistency after partial failures: if the database commits but the cache update fails, a pending outbox entry drives asynchronous repair.
- quote: "Record durable cache-update intent in the same SQL transaction as the business write. Use a SQL outbox table so a function crash after SQL commit doesn't lose the required Redis update. ... Make outbox processing durable and idempotent. Track pending, completed, failed, retry count, next retry time, and last error so the repair function can resume safely after failures."
- url: https://learn.microsoft.com/en-us/azure/architecture/databases/architecture/write-through-caching-azure-sql-managed-redis
- source_type: primary
- published: 2026-06-30
- confidence: high

### [8] Version-tagged cache entries (SQL rowversion, ETags) prevent stale repair operations from overwriting fresher cache data during write-through recovery.
- quote: "Use SQL rowversion, version numbers, timestamps, or ETags in cached values. Prevent older retries or repair jobs from overwriting newer cache entries."
- url: https://learn.microsoft.com/en-us/azure/architecture/databases/architecture/write-through-caching-azure-sql-managed-redis
- source_type: primary
- published: 2026-06-30
- confidence: high

### [9] Most distributed caches, including Redis, prioritize availability and partition tolerance over strong consistency (CAP theorem), meaning reads from cache replicas can return stale data during network partitions.
- quote: "The CAP theorem states that a distributed system can provide at most two of three guarantees: consistency, availability, and partition tolerance. Because network partitions are unavoidable in cloud environments, you must choose between consistency and availability. Most distributed caches, including Redis, prioritize availability and partition tolerance over strong consistency."
- url: https://learn.microsoft.com/en-us/azure/architecture/best-practices/caching
- source_type: primary
- published: 2026-03-25
- confidence: high

### [10] Redis provides multiple eviction policies; the default volatile-lru evicts only keys with a TTL set, while allkeys-lru applies LRU eviction globally — policy choice directly impacts cache hit ratio and memory utilization.
- quote: "When Redis reaches its memory limit, it evicts keys according to a configured eviction policy. The default policy is volatile-lru, which evicts the least recently used key that has a TTL set. Other policies include allkeys-lru, volatile-random, and noeviction."
- url: https://learn.microsoft.com/en-us/azure/architecture/best-practices/caching
- source_type: primary
- published: 2026-03-25
- confidence: high

### [11] Cache-aside implementations must avoid caching null values and should prime the cache with frequently accessed data at startup to minimize cold-start latency.
- quote: "Avoid caching a null value. ... Many solutions prepopulate the cache with data that an application likely requires as part of the startup processing. The Cache-Aside pattern remains useful when some of this data expires or gets evicted."
- url: https://learn.microsoft.com/en-us/azure/architecture/patterns/cache-aside
- source_type: primary
- published: 2025-09-11
- confidence: high

### [12] Redis replication is asynchronous, meaning a small amount of recently written data can be lost during an unexpected failover; for cache-layer durability this is generally acceptable since the database remains the authoritative store.
- quote: "Each Azure Managed Redis instance uses primary/replica replication. The service monitors node health and automatically promotes a replica if the primary fails. Because replication is asynchronous, a small amount of recently written data can be lost during an unexpected failover."
- url: https://learn.microsoft.com/en-us/azure/architecture/best-practices/caching
- source_type: primary
- published: 2026-03-25
- confidence: high
